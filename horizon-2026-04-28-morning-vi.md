## [The woes of sanitizing SVGs](https://muffin.ink/blog/scratch-svg-sanitization/) ⭐️ 6.0/10

Security researchers published a detailed case study examining the challenges of sanitizing SVG content to prevent XSS vulnerabilities on the Scratch platform, highlighting how regex-based approaches prove inadequate against sophisticated attack vectors. SVG sanitization remains a critical and often underestimated security concern for any platform allowing user-uploaded vector graphics. The Scratch case demonstrates that even platforms with attacker-controlled Node access face persistent XSS risks when proper defensive measures like CSP are not implemented. The article categorizes SVGs into two risk levels: simple 'bunch of paths with fills' versus 'clever dangerous stuff' with dynamic capabilities. Content Security Policy is identified as the only credible mitigation, and can be set directly in HTML using meta tags when HTTP headers aren't available, such as in iframe srcdoc scenarios.

hackernews · varun_ch · Apr 27, 15:31

**Background**: SVG files can contain embedded JavaScript, event handlers, and external resource references that execute in the browser context, making them a potent vector for cross-site scripting attacks. Platforms allowing user-uploaded SVGs must carefully sanitize these files to strip dangerous elements while preserving legitimate graphic functionality. Regex-based sanitization is particularly problematic because SVG's complex nested structure allows attackers to exploit parsing ambiguities and bypass pattern matching.
