## [Java Reified Generics Proposal to Fix Type Erasure](https://dev.to/luidmidev/java-generics-type-erasure-and-a-conceptual-idea-reified-generics-as-a-first-class-citizen-1dm6) ⭐️ 6.0/10

A technical article proposes introducing a 'reified' class modifier in Java that would maintain full runtime generic reification, coexisting with the existing type erasure system to solve the workaround-heavy ecosystem used by frameworks like Jackson, Hibernate, and Spring. Modern JVM frameworks currently resort to artificial type reconstruction using mechanisms like TypeReference, TypeToken, and ResolvableType. Reified classes could eliminate these workarounds entirely, enabling natural constructs like `Box<String>.class`, pattern matching with parameterized types, and direct reflection over type arguments. The proposal explicitly retains the legacy erasure-based system while adding a second reified class system with explicit, compiler-enforced boundaries between both worlds. Key technical challenges include cross-world inheritance scenarios such as whether a reified class can extend an erased one and how type assignability should work across the boundary.

rss · Dev.to · May 8, 05:35

**Background**: Java adopted type erasure in Java 5 (2004) for binary compatibility with the massive enterprise ecosystem. Unlike reified generics in languages like C#, Java discards generic type parameters at runtime, and no new classes are created for parameterized types. The OpenJDK Valhalla project has explored reification as a potential future enhancement, recognizing that reifying type information carries significant costs in both performance and complexity.
